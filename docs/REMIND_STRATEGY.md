# Remind Strategy

> "저장만 되고 잊혀지는 링크를, 다시 보게 만든다"는 제품의 핵심 가치를 실현하기 위한
> 리마인드 전략 설계 문서. PRD의 "링크 재노출" 기능과 Vercel Cron 기반 리마인드
> 데이터 흐름을 어떻게 구체화할지를 정의한다.

---

## 1. 왜 리마인드가 핵심인가

일반 북마크와 save-it의 차이는 "재노출"에 있다. 사용자가 직접 목록을 열어야만
링크를 다시 보는 구조라면 기존 북마크와 다를 바 없다. 따라서 시스템이 **적절한
시점에 적절한 링크를 사용자에게 다시 꺼내주는 것**이 제품의 본질이다.

리마인드 설계는 세 가지 질문으로 축약된다.

1. **무엇을** 다시 보여줄 것인가 (선정 알고리즘)
2. **언제** 보여줄 것인가 (트리거)
3. **어떻게** 보여줄 것인가 (채널 / UX)

---

## 2. 리마인드 대상 선정 (What)

DB 스키마(`links` 테이블)에서 활용 가능한 신호:

| 필드 | 의미 | 활용 |
|------|------|------|
| `is_read` / `read_at` | 읽음 여부와 시점 | 미열람 우선 노출 |
| `priority` | 중요도 | 가중치 상승 |
| `created_at` | 저장 시점 | 저장 후 경과일 기반 주기 리마인드 |
| `folder.para_category` | PARA 분류 | 분류별 다른 리마인드 정책 |

### 2.1 PARA 카테고리별 정책

PARA는 단순 폴더가 아니라 **"재노출 빈도"를 결정하는 기준**으로 활용한다.

| 카테고리 | 리마인드 성격 | 기본 정책 |
|----------|---------------|-----------|
| **Projects** | 진행 중 → 자주 | 미열람 시 1~3일 내 재노출, 우선순위 최상 |
| **Areas** | 지속 관심 → 중간 | 주 1회 랜덤 샘플링 |
| **Resources** | "나중에" → 약하게 | 2~4주 경과 미열람 시 1회 재노출 |
| **Archives** | 완료/비활성 | 리마인드 제외 (검색으로만 접근) |
| **미지정** | 기본 폴더 | Resources와 동일 정책 적용 |

### 2.2 선정 점수 (Scoring)

단일 규칙이 아닌 **가중치 합산 점수**로 선별한다. 점수가 높은 상위 N개만
리마인드 후보로 올린다.

```
score =
  w1 * (미열람 여부: 0/1)
+ w2 * (priority 정규화: 0~1)
+ w3 * (PARA 카테고리 가중치)
+ w4 * (저장 후 경과일 기반 감쇠 함수)
- w5 * (직전 리마인드 후 경과일이 짧으면 감점 → 리마인드 피로 방지)
```

- 경과일 감쇠는 "너무 최근"과 "너무 오래됨" 둘 다 점수를 낮추는 종 모양
  곡선을 쓰면, 저장 직후에는 당장 노출하지 않고 며칠 뒤 최적점에서 노출된다.
- 리마인드 이력을 추적하려면 별도 테이블이 필요하다 (§6 스키마 확장 참고).

### 2.3 선정 모드 (후보 묶음)

한 번의 리마인드에 담는 링크 묶음은 용도가 섞이지 않도록 모드별로 분리한다.

- **"다시 볼 시간" 모드** — 미열람 + 경과일 최적
- **"중요한데 놓친 것" 모드** — priority 상위 + 미열람
- **"오래된 Resource 재발굴" 모드** — Resources 중 30일+ 미열람
- **"읽었지만 복습할 만한 것" 모드** — 읽음 + priority 상위 (선택 기능)
- **"같은 종류 콘텐츠" 모드 (컨텍스트 매칭)** — 현재 보고 있는 사이트와
  같은 도메인/콘텐츠 타입의 저장 링크 노출. §3.3 참고.

## 3. 리마인드 트리거 (When)

### 3.1 Scheduled (Push) 리마인드

Vercel Cron → Next.js API Route → Supabase 경로(ARCHITECTURE.md §데이터 흐름)로
주기적으로 돌며 후보를 산출해 사용자에게 전달한다.

- **일일 다이제스트** — 하루 1회, 아침 9시 (사용자 타임존 기준). 3~5개.
- **주간 회고** — 주 1회, 일요일 저녁. 이번 주 저장/미열람 요약.
- 사용자별로 시간/요일은 설정 가능해야 한다. 고정하면 무시되기 시작한다.

### 3.2 In-Session (Pull) 리마인드

사용자가 제품에 들어왔을 때 노출되는 패시브 리마인드. 알림 피로 없이 가장
자연스럽다.

- **홈/대시보드 진입 시** — "오늘 다시 볼 링크" 섹션
- **새 링크 저장 직후** — "관련 폴더에 미열람 N개가 있어요" 힌트
- **확장 프로그램 팝업** — 현재 탭의 도메인과 같은 링크가 이미 저장돼 있으면 안내

### 3.3 Contextual — 콘텐츠 매칭 모드

사용자가 **지금 어떤 콘텐츠를 소비 중인지**를 단서로 같은 종류의 저장 링크를
꺼내 보여주는 방식. 익스텐션이 가장 강점을 가지는 영역이며, 다른 어떤
리마인드보다 "다시 보게 만든다"는 가치에 직결된다.

저장 행위 자체가 "이런 콘텐츠가 좋다"는 신호이므로, 같은 맥락에 들어왔을 때
다시 꺼내주면 사용자가 의식적으로 결정하지 않아도 자연스럽게 재소비된다.

#### 우선 케이스: YouTube 모드

가장 명확한 컨텍스트는 YouTube다. 사용자가 영상을 한 편 보고 나면 다음 영상을
찾는 행동이 이어지는데, 이때 알고리즘 추천 대신 **본인이 이전에 저장한 영상**을
보여주면 강력하다.

- **트리거**: 익스텐션이 활성 탭의 URL이 `youtube.com/watch` (또는
  `youtu.be`, `youtube.com/shorts`) 임을 감지
- **동작**: 익스텐션 팝업/사이드 패널에 "내가 저장한 YouTube 영상" 섹션 표시
  - 미열람 우선
  - 같은 채널 영상이 있으면 가중치 상승 (URL 또는 메타데이터 기반)
  - 썸네일 + 제목 + 저장 폴더(PARA) 라벨로 카드 노출
- **선택 확장**: 영상 페이지에 인라인 위젯을 주입해 "관련 저장 영상"을
  YouTube 추천 영역 옆에 노출 (Content Script). 침습적이므로 옵션으로 제공.

#### 일반화: 도메인/콘텐츠 타입 매칭

YouTube를 시작점으로, 다른 패턴으로도 확장한다.

| 컨텍스트 | 매칭 규칙 | 보여줄 묶음 |
|----------|-----------|-------------|
| YouTube `watch` 페이지 | 호스트 = `youtube.com` | 저장된 YouTube 영상 |
| 블로그/뉴스 글 | URL이 article로 판별됨 | 저장된 글 형태 링크 |
| GitHub 레포 | 호스트 = `github.com` + path 패턴 | 저장된 GitHub 링크 |
| 동일 도메인 일반 | 호스트 일치 | 같은 도메인 저장 링크 |
| 동일 채널/저자 | 메타데이터 일치 | 같은 출처 링크 |

콘텐츠 타입 분류는 단순한 호스트 매칭으로 시작해도 충분하다. 메타데이터 기반
분류는 v2 이후 도입.

#### 구현 메모

- 익스텐션의 활성 탭 URL은 `chrome.tabs` API로 즉시 조회 가능 → 매칭 자체는
  네트워크 호출 없이 가능.
- 후보 산출은 **Supabase 직접 호출** (ARCHITECTURE.md 데이터 흐름 그대로).
  특정 호스트나 URL 접두어로 필터링하는 쿼리만 추가하면 된다.
- 검색 효율을 위해 `links` 테이블에 `host text generated always as (...) stored`
  같은 파생 컬럼이나 함수 인덱스를 두는 것을 검토.

#### 기타 컨텍스트 (장기)

- 캘린더/업무 컨텍스트 연동 (예: "회의 전, 관련 Project 링크")
- 시간대 기반 (예: 출근길 시간대에 "짧게 볼 영상")

## 4. 전달 채널 (How)

| 채널 | 장점 | 단점 | 권장 시점 |
|------|------|------|-----------|
| **웹 대시보드** | 무료, 거부감 낮음 | 사용자가 들어와야 봄 | MVP (Pull) |
| **크롬 익스텐션 뱃지/팝업** | 브라우저 사용 중 자연 노출 | Chrome 한정 | MVP (Pull) |
| **이메일** | 이탈 사용자 재방문에 효과 | 피로 쉬움 | v1 (일일/주간 다이제스트) |
| **웹 푸시 알림** | 즉시성 | 권한 허용률 낮음 | v2 이후 |
| **모바일 푸시** | 강력 | 앱 필요 | 로드맵 밖 |

MVP 기준으로는 **Pull 2종 (대시보드 + 확장)** 부터 구현하고, 이메일 다이제스트를
v1에서 추가한다. Push 알림은 지표상 필요해지면 붙인다.

## 5. UX 원칙

리마인드는 "불러내기"와 "밀어내기" 사이의 균형이다. 밀리면 광고처럼 느껴지고,
약하면 기능이 없는 것과 같다.

1. **행동 한 번에 해결** — 리마인드 카드에서 열기/보류/Archive 이동을 한 번에.
2. **"왜 보여주는가"를 노출** — "7일 전 저장, 아직 안 읽음" 같은 이유 라벨을
   함께 보여주면 신뢰도가 오른다.
3. **스누즈(미루기) 제공** — "3일 뒤 다시"를 선택할 수 있으면 거부감이 줄고
   리마인드 이력이 신호로 쌓인다.
4. **일괄 작업** — "이번 묶음 모두 Archive" 같은 청소 액션을 제공해야
   Resources가 부풀어도 관리 비용이 낮다.
5. **빈도 상한** — 하루 1회 이상 Push 리마인드를 보내지 않는다. 사용자가
   원하면 상향 조정.

## 6. 스키마 확장 제안

현재 `links` 테이블만으로는 "같은 링크를 너무 자주 리마인드하지 않기"를
보장할 수 없다. 최소한의 이력 추적이 필요하다.

### 6.1 `link_reminders` (리마인드 이력)

```sql
create table link_reminders (
  id uuid primary key default gen_random_uuid(),
  link_id uuid not null references links(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  channel text not null,       -- 'dashboard' | 'extension' | 'email' | ...
  mode text not null,          -- 'daily' | 'weekly' | 'resurface' | ...
  sent_at timestamptz not null default now(),
  opened_at timestamptz,       -- 사용자가 링크를 실제로 열었는지
  snoozed_until timestamptz    -- 스누즈 지정 시각
);

create index idx_link_reminders_link on link_reminders(link_id, sent_at desc);
create index idx_link_reminders_user on link_reminders(user_id, sent_at desc);
```

- 점수 계산 시 `sent_at` 최신값으로 피로도 감점을 계산한다.
- `opened_at` 집계로 리마인드 채널/모드별 CTR(열람률)을 계측한다.

### 6.2 `user_reminder_prefs` (개인화 설정)

```sql
create table user_reminder_prefs (
  user_id uuid primary key references auth.users(id) on delete cascade,
  daily_enabled boolean not null default true,
  daily_time time not null default '09:00',
  timezone text not null default 'Asia/Seoul',
  weekly_enabled boolean not null default true,
  email_enabled boolean not null default false,
  max_items_per_reminder smallint not null default 5
);
```

## 7. 알고리즘 구현 위치

ARCHITECTURE.md의 데이터 흐름을 그대로 따른다.

```
Vercel Cron (hourly or daily)
  → Next.js API Route  (/api/cron/remind)
    → Supabase RPC or SQL
      → 사용자별 후보 산출
      → link_reminders insert
      → 채널별 전달 (UI는 대시보드가 읽고, 이메일은 Resend 등으로 발송)
```

- 후보 산출 로직은 **DB 함수(PL/pgSQL)** 로 두는 편이 네트워크 왕복이 적다.
- 이메일 발송은 별도 서비스(Resend/Postmark) 연동이 필요하면 v1에서 도입.

## 8. 단계별 로드맵

### MVP (현재 스키마 + 최소 구현)
- 대시보드 "오늘 다시 볼 링크" 섹션 (Pull)
  - 규칙: 미열람 + priority 내림차순 + 저장 후 3일 이상, 상위 5개
- 확장 프로그램 팝업에 "미열람 N개" 표시
- **익스텐션 YouTube 모드 (§3.3)** — 활성 탭이 YouTube `watch`일 때
  팝업에 저장된 YouTube 영상 노출. 호스트 매칭만으로 구현 가능하므로
  MVP에 포함할 가치가 충분히 크다.

### v1 (리마인드 이력 도입)
- `link_reminders`, `user_reminder_prefs` 테이블 추가
- Vercel Cron으로 일일 다이제스트 이메일 발송
- PARA 카테고리별 정책 반영한 스코어링 도입
- 스누즈 기능
- 컨텍스트 매칭 일반화 — 일반 도메인 매칭, GitHub/블로그 등 콘텐츠 타입 확장
- (선택) YouTube 페이지 내 인라인 위젯 (Content Script)

### v2 (개인화 / 실험)
- 주간 회고 이메일
- 채널/모드별 CTR로 가중치 자동 튜닝 (A/B)
- 웹 푸시 알림
- 메타데이터 기반 채널/저자 매칭, 시간대/캘린더 컨텍스트

## 9. 성공 지표 매핑

PRD §6의 KPI를 리마인드 관점에서 구체화한다.

| PRD KPI | 리마인드 측정 지표 |
|---------|--------------------|
| 저장된 링크 재열람률 | `link_reminders.opened_at` 비율 |
| 저장 후 7일 내 클릭 비율 | `created_at` 기준 7일 내 `opened_at` 발생 비율 |
| 링크 소비 완료율 | 리마인드 → `links.is_read = true` 전환율 |
| 사용자 재방문율 | 이메일/푸시 리마인드에서 유입된 세션 비율 |

채널/모드별로 쪼개서 보면, 어떤 리마인드가 실제로 "다시 보게 만드는지"
판별할 수 있다. 이 데이터가 v2의 가중치 튜닝 재료가 된다.

## 10. 열린 질문

- Archives는 정말로 리마인드에서 완전히 배제해야 하는가, 아니면 연 1회
  "과거에 중요했던 자료" 같은 장기 리마인드로 활용할 수 있는가?
- PARA 분류를 사용자가 안 하면 리마인드 품질이 급락한다. 자동 분류 추천
  (PRD §5 권장 기능)과 리마인드 중 어느 쪽을 먼저 붙일지 순서를 정해야 함.
- 리마인드 피로 방지 가중치는 하드코딩할지, 사용자가 직접 "리마인드 강도"를
  조절하게 할지 결정 필요.
